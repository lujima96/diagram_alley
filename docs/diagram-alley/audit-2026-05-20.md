## 2. New Issues Found — All Fixed in This Audit

### Issue 1 — F5 §9.1: 6 error codes from D1 and D5 not added

**Problem:** D1 defined `PROVIDER_NOT_CONFIGURED`, `PROMPT_TOO_LONG`, `GENERATION_PARSE_FAILED`, `GENERATION_INVALID_SPEC`, and `PROVIDER_TIMEOUT` but did not patch F5. D5 defined `IMPORT_UNSUPPORTED_FORMAT` but did not patch F5. The session summary acknowledged these were "tracked in D1 spec itself, not as a direct F5 edit."

**Fix:** Added all 6 codes to F5 §9.1:
- `400 PROVIDER_NOT_CONFIGURED` — No provider configured for this user
- `400 PROMPT_TOO_LONG` — Prompt or change request exceeds length limit
- `400 IMPORT_UNSUPPORTED_FORMAT` — Import format or Mermaid subtype not supported in V1
- `422 GENERATION_PARSE_FAILED` — LLM response is not parseable as JSON after extraction fallback
- `422 GENERATION_INVALID_SPEC` — LLM spec fails validation after repair attempt
- `503 PROVIDER_TIMEOUT` — Provider response took >30s (per-request timeout)

Also updated `502 PROVIDER_UNAVAILABLE` description to clarify "after 1 retry".

### Issue 2 — D2 §2.4: Subscription gate not documented in PATCH steps

**Problem:** D7 §10.2 noted that `PATCH /api/v1/diagrams/{id}` must block `spec_json` updates for expired users (`status='canceled'`), stating "D2 should be read as incorporating this constraint." This was a soft cross-spec note, not an explicit documented step in D2 §2.4.

**Fix:** Added as step 1 in the D2 §2.4 `spec_json` processing sequence: check `subscription.status`; if `'canceled'`, return `403 SUBSCRIPTION_REQUIRED`. Title and `project_id` updates remain permitted. References D7 §2.3.

### Issue 3 — D3 §4.1: Pre-restore snapshot not in service steps

**Problem:** D3 §8 (Version History UI) described a "Before restore to <timestamp>" snapshot created of the current spec before any restore. This was not in the service steps in D3 §4.1, which only documented the post-restore snapshot ("Restored from version {short_vid}"). Every restore should produce 2 new version rows, but §4.1 only specified 1.

**Fix:** Added step 5 to D3 §4.1: "Create a pre-restore snapshot of the current `diagrams.spec_json` with `change_summary = 'Before restore to <version_created_at timestamp>'`." Steps 5–11 renumbered to 6–12.

### Issue 4 — F1 §5.3: Misleading "automatic snapshots" wording

**Problem:** F1 §5.3 said "The service layer debounces auto-saves at minimum 30s between automatic snapshots." This implies auto-saves create snapshots, which contradicts D2 §5.3 and D3 §1 (snapshots only on explicit Save, AI modify accepted, or restore — never on auto-save).

**Fix:** Replaced with: "A version is not created on every keystroke, every render call, or on auto-save. Auto-save (D2 §5.3) fires after 30 seconds of inactivity and persists the spec via PATCH without creating a version row."
