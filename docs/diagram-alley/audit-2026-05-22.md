# Diagram Alley — Planning Corpus Audit
**Date:** 2026-05-22
**Scope:** Phase 6 post-mockup audit — cross-references 16 specs (F0–F5, D1–D7, A1), decisions-log.md, and 21 HTML mockups

---

## Executive Summary

The planning corpus is **largely coherent** with very few contradictions, but there are **strategic gaps and naming inconsistencies** that should be resolved before implementation begins. The data model (F1) and API contract (F5) are solid and internally consistent. Primary gaps are missing visual specs for complex workflows (AI Improve, Inspector tabs, error states) and a completely unmocked Admin Console.

---

## 1. Contradictions (Mockup vs. Spec)

### 1.1 Trial Expiry — UI Lock Level vs. API Gate

**Finding:** D7 §2.3 and D2 §2.4 say the backend blocks edits via 403 SUBSCRIPTION_REQUIRED at the API layer. However, **trial-expired.html** shows the visual editor itself locked with an overlay, suggesting UI-level lockout.

**Impact:** Mid. The specs prescribe API-level gating; the mockup shows UI-level lockout. The correct approach (confirm before implementing) is to lock at the API layer and reflect that in the UI state.

**Spec:** D2 §2.4, F3 §5.5, D7 §2.3 | **Mockup:** desktop/trial-expired.html

---

### 1.2 Sidebar Navigation Inconsistency

**Finding:** dashboard.html shows "Providers" in the sidebar footer with an avatar showing "Pro · 12 days left". editor.html shows "Trial · 12 days left" on the avatar. F4 §3.2 defines the sidebar as having a Settings section with sub-items (Account, Billing, Providers). Some mockups render these consistently; others don't.

**Impact:** Minor. UX consistency issue. The sidebar settings sub-menu should be uniform across all authenticated pages.

**Spec:** F4 §3.2 | **Mockup:** dashboard.html, editor.html, billing.html

---

### 3.5 Error State UX

**Finding:** F5 §9.1 defines error codes (PROVIDER_UNAVAILABLE, RATE_LIMITED, SPEC_VALIDATION_FAILED, SUBSCRIPTION_REQUIRED, etc.). F5 §9.2 defines frontend responses (e.g., redirect to `/settings/billing` with upgrade banner). **No mockups show any error state** (validation failed banners, provider unavailable states, rate limit messages).

**Impact:** Mid. Error UX is critical for user trust; none of it is mocked.

**Recommendation:** Create at minimum one error state mockup covering SPEC_VALIDATION_FAILED, PROVIDER_UNAVAILABLE, and SUBSCRIPTION_REQUIRED.

**Spec:** F5 §9.1, F5 §9.2

### 3.6 Inspector Panel Tabs (Selection, Validation, Spec Editor)

**Finding:** F4 §5.5 defines three inspector tabs — Selection (editable node/edge properties), Validation (live error list, warning list, auto-fix button), and Spec Editor (CodeMirror 6 JSON escape hatch). editor.html shows only a placeholder ("No node selected · Click a node to inspect"). **No mockup shows the three tabs in detail.**

**Impact:** Mid. The inspector is one of the most complex UI components and is entirely unspecified visually.

**Recommendation:** Create detailed inspector mockups, especially for the Validation tab and Spec Editor tab.

**Spec:** F4 §5.5

---

### 3.7 AI Improve Proposal and Diff Display

**Finding:** D1 §4.2 and D1 §4.3 define the AI improve workflow: send current spec + change request → receive proposed spec + diff metadata → frontend displays diff and lets user accept or reject. **No mockup shows this flow.**

**Impact:** HIGH. AI modification is a core feature and entirely unvisually specified.

**Recommendation:** Create a mockup showing: (1) user fills Improve field and clicks Improve, (2) loading state, (3) diff preview (nodes_added, nodes_removed, nodes_changed), (4) Accept/Reject buttons, (5) result after accept (version snapshot created).

**Spec:** D1 §4.2, D1 §4.3, D3 §7

**Recommendation:** Define the archived project UX before implementing project CRUD.

**Spec:** F1 §4.2, F4 §2

### 4.2 User-Created Templates in V1

**Finding:** F1 §4.5 defines `templates.is_public` as always false in V1 (marketplace deferred). Specs do not clearly state whether V1 users can create their own private templates or only system templates exist.

**Recommendation:** Resolve in D2 or create a templates decision log entry.

**Spec:** F1 §4.5

### 4.3 Diagram-to-Project Move Operations

**Finding:** F4 §2 defines project routes and F5 §10 defines project CRUD, but specs do not define whether diagrams can be moved between projects.

**Impact:** Low. Schema allows it; workflow is not specified.

**Spec:** D7 §6, D7 §8, F1 §5.2

**Recommendation:** Define grid editor appearance for these diagram types in F2 and F4 before implementing their editors.

**Spec:** F2 §6.3, F4 §6.4, F1 §3.4, F1 §3.5

## 5. Schema / Data Model Issues

### 5.1 User Display Name Fallback

**Finding:** F1 §4.1 defines `users.name` as NULLABLE. Mockups show user initials and display names throughout (e.g., "Priya R." in dashboard.html). Specs do not define the fallback when name is NULL.

**Impact:** Low. Define the fallback (use email prefix? "User"?) in F1 or F3 before implementing the sidebar.

**Spec:** F1 §4.1 | **Mockup:** dashboard.html, editor.html, trial-expired.html

---

### 5.2 Subscription State Ambiguity

**Finding:** F1 §5.2 lists subscription states (trialing, active, past_due, canceled, incomplete). D7 §2.3 says "On trial expiry (`status = 'canceled'`)" — this conflates a canceled trial with a canceled Pro subscription. The combination of `plan = 'trial'` and `status = 'canceled'` means expired trial, but this isn't stated explicitly.

**Impact:** Low. Add a clarifying note to D7 §2.3 to avoid implementation confusion.

**Spec:** F1 §5.2, D7 §2.3

---

## 6. Naming Inconsistencies

### 6.1 "Strict ASCII" vs "ASCII" Tab Labels

**Finding:** The export modal correctly shows a "Strict ASCII / UTF-8 Text" toggle (per DEC-023). However, preview tabs in editor.html and share-view.html just say "ASCII" without distinguishing modes.

**Recommendation:** "ASCII" is acceptable as a tab label (implies Strict ASCII by default). Only show the mode toggle in export settings where the choice matters. Codify this in D4 or design-system.md.

**Spec:** DEC-023, F2 §4.1 | **Mockup:** desktop/editor.html, desktop/export-modal.html, desktop/share-view.html

---

### 6.2 "Improve" vs "Improve existing"

**Finding:** editor.html uses `[Improve]` button. F4 §5.2 says `[Improve existing]`. D1 calls the flow "AI modification" or "AI improve".

**Recommendation:** Standardize to `[Improve]` across all specs and mockups.

**Spec:** F4 §5.2, D1 §4 | **Mockup:** desktop/editor.html

---

### 6.3 "Dirty" State Terminology

**Finding:** D2 §5.1 defines two flags: `has_unpersisted_changes` (in-memory vs. last save) and `has_unversioned_changes` (live save vs. latest version). Some specs say "unsaved changes"; others say "unversioned changes". The distinction matters for the toolbar save indicator.

**Recommendation:** Codify both terms in D2 §5.1 and ensure the toolbar mockup and spec use consistent language.

**Spec:** D2 §5.1, D2 §5.2 | **Mockup:** components/toolbar.html
